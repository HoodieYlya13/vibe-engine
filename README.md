# Vibe Engine (Simulation Core) 🦀

**Status:** 🚧 In Active Development

A dedicated physics and logic engine built with **Rust** and compiled to **WebAssembly (WASM)** via `wasm-pack`. This engine powers the **WebGPU-accelerated** _Vibe Heist_ project, handling performant 3D physics calculations off the main JavaScript thread.

## ⚡ Tech Stack

- **Language:** Rust
- **Physics:** Rapier3D
- **Math:** Nalgebra
- **Compilation:** WASM-Bindgen / `wasm-pack`

## 🔧 Build Instructions

**Prerequisites:** Rust, Cargo, `wasm-pack`.

```bash
# Emit the shared layout constants for the client
# (writes ../client/src/generated/sim-layout.ts)
cargo run --bin gen_layout

# Compile to WebAssembly (emits pkg/)
wasm-pack build --target web
```

## 🧪 Tests

The crate also builds natively (`rlib`), so the deterministic replay tests run as plain `cargo test` — no browser or WASM needed:

```bash
cargo test
```

These drive the vehicle (`tests/drive.rs`: drive, brake-to-reverse, handbrake, steering, sleep/wake), the on-foot player (`tests/walk.rs`: walk, run, walls, jump, ramp, car-chassis blocking), the handling assists (`tests/assists.rs`: rear-steer drift, reverse steering damping, flip prevention/self-righting, airborne control, handbrake tail-out slip — each compared against its disabled variant), the vehicle classes (`tests/classes.rs`: data-file parse, tuning round-trip, per-class acceleration ordering, pose-preserving class switch), and the impact-damage model (`tests/damage.rs`: damage scales with crash speed, clean driving costs nothing, zero health kills the drivetrain with the explosion pop, reset repairs) through scripted input sequences and assert on the resulting positions, plus bitwise determinism checks. They are the regression guard for physics changes — run them before pushing.

CI runs `cargo fmt --check`, `cargo clippy --all-targets -- -D warnings`, and `cargo test` on every push.

## 📂 Source Layout

```
data/
  vehicles.ron   — vehicle classes (sedan/sports/truck): chassis, wheels,
                   drive tuning, impact-damage spec — the source of truth
                   for how a class drives (and breaks)
src/
  lib.rs         — module wiring, re-exports SimEngine
  engine.rs      — SimEngine struct, constructors, fixed-step system ordering
  constants.rs   — single source of truth (input bits, SAB layout, world geometry)
  bridge/        — the sim ⇄ client boundary
    snapshot.rs  —   state writer (car, player, HUD, camera, peds)
    input.rs     —   button decoding
    controls.rs  —   reset / pause / sleep queries
    dev.rs       —   dev console surface: live tuning, teleports, flip-car,
                     damage-car/health, debug wireframes
  world/         — static colliders (arena, greybox blocks + ramps)
  vehicle/       — ray-cast vehicle controller + runtime VehicleTuning
                   (incl. arcade assists: anti-roll righting, rear steer,
                   reverse steer scaling, airborne control, handbrake
                   rear-grip cut for drifting) + chassis impact damage
                   (solver-step Δv → health, explosion pop + dead drivetrain
                   at zero); spec.rs loads the classes from data/vehicles.ron
                   (embedded at build)
  character/     — kinematic character controller (player)
  crowd/         — decorative ped simulation (Phase 4 rewrites this)
```

## 🎯 Architecture

This module is strictly decoupled from the rendering layer. It steps the world at a fixed timestep in a Web Worker, reads inputs from a shared ring buffer, and writes snapshots into a double-buffered `SharedArrayBuffer`, ensuring the simulation remains deterministic and framework-agnostic.

The car is a Rapier `DynamicRayCastVehicleController`; the on-foot player is a kinematic body + capsule driven by Rapier's `KinematicCharacterController` against the same colliders — world geometry is defined exactly once.

`src/constants.rs` is the **single source of truth** for input bits, SAB layout offsets/strides, and greybox world geometry. The TypeScript mirror (`client/src/generated/sim-layout.ts`) is generated by `cargo run --bin gen_layout` — never edit it by hand; change the constants and regenerate.
